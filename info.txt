--------------------------------------------------------------------------------
PHASE 1: PROJECT OVERVIEW & MISSION STATEMENT
--------------------------------------------------------------------------------
The Digital Wallet application is a modern financial management tool designed to
help users track their savings, manage expenses, and visualize their financial
health. The mission of this project is to provide a seamless, secure, and
intuitive platform for personal finance management.

Key Features:
- User Authentication (Signup/Login)
- Savings Management
- Real-time Expense Tracking
- Visual Data Representation (Charts)
- E-commerce Integration (Product List & Cart)
- User Profile Management

The project follows a decoupled architecture with a dedicated Node.js/Express
backend and an Angular 19 frontend, ensuring scalability and ease of maintenance.

PHASE 3: FULL ARCHITECTURE EXPLANATION (FRONTEND + BACKEND)
--------------------------------------------------------------------------------
The application follows a Client-Server Architecture.

1. Frontend (Client Layer):
   - Built with Angular, it serves as the user interface.
   - It communicates with the backend via RESTful APIs using Angular's HttpClient.
   - State is managed locally (LocalStorage) for authentication tokens and user
     details.
   - Components are modularized, each handling a specific part of the UI.

2. Backend (Server Layer):
   - Built with Express, it handles business logic and database interactions.
   - It exposes endpoints for authentication and financial data management.
   - Middleware is used for request parsing, CORS, and JWT authentication.

3. Database (Data Layer):
   - MongoDB stores user profiles, savings records, expenses, and cart items.
   - Mongoose provides a schema-based solution to model application data.

4. Communication:
   - All data exchange happens via JSON over HTTP.
   - Security is enforced via JWT headers for protected routes.

PHASE 6: API REFERENCE & COMMUNICATION PROTOCOLS
--------------------------------------------------------------------------------
The backend exposes the following RESTful endpoints:

AUTH ENDPOINTS:
- POST /signup:
    - Body: { username, email, password }
    - Description: Registers a new user.
- POST /login:
    - Body: { email, password }
    - Response: { token, userName, userEmail }
    - Description: Authenticates user and returns a JWT.

DATA ENDPOINTS (Requires Authorization Header):
- POST /api/savings:
    - Body: { amount }
    - Header: Authorization: Bearer <token>
    - Description: Records a new saving entry for the authenticated user.
- POST /api/expenses:
    - Body: { category, amount }
    - Header: Authorization: Bearer <token>
    - Description: Records a new expense.

MISC ENDPOINTS:
- POST /api/cart:
    - Body: Array of cart items.
    - Description: Saves cart data to the database.

All responses are returned in JSON format with appropriate HTTP status codes:
- 200 OK: Successful request.
- 201 Created: Resource successfully created.
- 400 Bad Request: Invalid input data.
- 401 Unauthorized: Invalid or missing token.
- 500 Internal Server Error: Unexpected server failure.

PHASE 8: SECURITY IMPLEMENTATION (AUTH, JWT, BCRYPT)
--------------------------------------------------------------------------------
Security is a top priority in this application.

1. Password Hashing:
   - The application uses `bcryptjs` with a salt factor of 10.
   - Raw passwords are never stored in the database.
   - During login, the provided password is compared with the hashed version
     using `bcrypt.compare()`.

2. Token-Based Authentication:
   - Upon successful login, the server generates a JSON Web Token (JWT).
   - The token contains the user's ID as a payload.
   - Tokens expire after 1 hour to limit exposure in case of theft.

3. Protected Routes:
   - The `authenticateJWT` middleware on the backend intercepts requests to
     private endpoints.
   - It verifies the token's validity and extracts the `userId`.
   - If the token is invalid or missing, a 401/403 error is returned.

4. CORS Configuration:
   - The backend specifically allows requests from `http://localhost:4200`
     (the Angular dev server).
   - This prevents unauthorized domains from accessing the API.

PHASE 10: BACKEND LOGIC & BUSINESS RULES
--------------------------------------------------------------------------------
The backend is the "brain" of the application.

Key Business Rules:
- Email Uniqueness: The database schema enforces unique emails for users.
- Auth Enforcement: Most data endpoints require a valid JWT.
- Input Validation: The server checks if amounts are valid numbers and categories
  are provided before saving to the database.
- Data Integrity: Savings and Expenses are linked to specific User IDs via
  MongoDB ObjectIds, ensuring users only see their own data.

The backend uses a simple procedural flow within route handlers, making the
logic easy to follow for beginners while remaining functional for small-scale
production use.

PHASE 11: STATE MANAGEMENT & DATA FLOW
--------------------------------------------------------------------------------
Instead of using heavy libraries like NgRx, this project uses a combination of:
- Services: To share logic and data between components.
- LocalStorage: To persist user authentication state across page refreshes.
- Component Properties: To manage local UI state (e.g., current inputs).

Data Flow:
1. User enters data in a component.
2. Component calls an Angular service (or HttpClient directly).
3. Service sends an HTTP request to the Express server.
4. Server interacts with MongoDB.
5. Server sends a response back to the service.
6. Service returns an Observable to the component.
7. Component updates its local state, triggering an Angular change detection
   cycle that updates the DOM.

PHASE 16: INTERVIEW QUESTIONS & DETAILED ANSWERS (EXPANDED)
--------------------------------------------------------------------------------
Q1: What is JWT and why do we use it in this project?
A: JWT stands for JSON Web Token. It is used for stateless authentication.
   The server generates a token after login, and the client sends it in the
   header of subsequent requests. This avoids storing session data on the server.
   In this project, we use `jsonwebtoken` library to sign and verify tokens.

Q2: How does Mongoose help in this project?
A: Mongoose provides a schema for MongoDB documents, allowing us to define
   data types and validation rules. It also provides powerful helper methods
   for querying the database. We define User, Saving, and Expense schemas
   using Mongoose to maintain structure in our NoSQL database.

Q3: Explain the benefit of using standalone components in Angular.
A: Standalone components reduce the need for `NgModules`, making the code
   cleaner and easier to understand. They simplify the dependency injection
   system and make components more portable. They were introduced in Angular 14
   and are now the standard in Angular 19.

Q4: How do you handle Cross-Origin Resource Sharing (CORS) in Express?
A: We use the `cors` middleware and configure the `origin` property to allow
   requests only from the frontend's URL. This is crucial for security as it
   prevents malicious sites from making requests to our API.

Q5: What is the purpose of Bcrypt.js?
A: Bcrypt.js is used for hashing passwords. It uses a slow hashing algorithm
   (Blowfish) and incorporates a salt to protect against rainbow table attacks.
   In our signup route, we hash the user's password before saving it.

Q6: How does the `HttpClient` in Angular work?
A: The `HttpClient` is a built-in service that allows Angular applications to
   communicate with backend services over HTTP. It returns Observables, which
   we can subscribe to in our components to handle the response.

Q7: What is the role of the `ngOnInit` lifecycle hook?
A: `ngOnInit` is called by Angular after the component's data-bound properties
   have been initialized. It is the best place to perform initialization logic,
   such as fetching data from a service.

Q8: Explain the difference between `localStorage` and `sessionStorage`.
A: `localStorage` persists data even after the browser is closed and reopened,
   whereas `sessionStorage` clears data when the page session ends (e.g., when
   the tab is closed). We use `localStorage` for our JWT token.

Q9: What are Observables in RxJS?
A: Observables are a way of handling asynchronous data streams. They allow
   multiple values to be emitted over time. In this project, `HttpClient`
   methods return Observables that we subscribe to.

Q10: How do you handle environment variables in a Node.js application?
A: Typically, we use a `.env` file and the `dotenv` package to store secrets
   like database URLs and API keys. This keeps sensitive information out of the
   source code.

PHASE 17: SYSTEM DESIGN DISCUSSIONS (EXPANDED)
--------------------------------------------------------------------------------
Topic: Scalability of the Digital Wallet
The current design is monolithic (single backend file). To scale:
1. Microservices: Separate authentication, financial tracking, and e-commerce
   into different services. This allows each service to be scaled independently
   based on its specific load.
2. Load Balancing: Use Nginx or AWS ELB to distribute traffic across multiple
   instances of the backend server. This prevents any single server from
   becoming a bottleneck.
3. Caching: Use Redis to cache frequently accessed data like user profiles or
   product lists. Caching significantly reduces the load on the database.
4. Database Sharding: As the user base grows, shard the MongoDB database to
   distribute data across multiple servers. Sharding helps maintain performance
   even with millions of records.
5. Message Queues: Implement a message queue like RabbitMQ or Kafka for
   asynchronous tasks like sending notification emails or processing large
   reports.

Topic: Security Hardening
1. Rate Limiting: Implement rate limiting on sensitive endpoints (like login)
   to prevent brute-force attacks.
2. Input Sanitization: Use libraries to sanitize user input to prevent XSS
   and NoSQL injection attacks.
3. HTTPS: Ensure all communication is encrypted using SSL/TLS certificates.
4. Monitoring: Set up real-time monitoring and alerting for unusual traffic
   patterns or server errors.

1. JWT Authentication Middleware (server.js):
```javascript
const authenticateJWT = (req, res, next) => {
  const authHeader = req.headers['authorization'];
  if (!authHeader) return res.status(403).json({ message: "No token provided" });

  const token = authHeader.replace('Bearer ', '');
  try {
    const decoded = jwt.verify(token, JWT_SECRET);
    req.userId = decoded.id;
    next();
  } catch (err) {
    return res.status(401).json({ message: "Invalid or expired token" });
  }
};
```
Explanation: This function checks for a 'Bearer' token in the Authorization
header. If valid, it attaches the user ID to the request object and calls
`next()`, allowing the request to proceed to the route handler.

2. Saving an Expense (sae.component.ts):
```typescript
addExpense(): void {
  // ... validation ...
  const category = this.selectedCategory;
  const amount = this.expenseAmountInput;

  this.savings -= amount;
  this.totalExpenses += amount;
  this.expenses.push({ category, amount });

  const token = localStorage.getItem("token");
  const headers = new HttpHeaders({
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${token}`
  });

  this.http.post('http://localhost:5001/api/expenses', { category, amount }, { headers })
    .subscribe({
      next: (res) => console.log('Expense added:', res),
      error: (err) => console.error('Error adding expense:', err)
    });
}
```
Explanation: This method handles the UI update immediately (optimistic update)
and then sends a POST request to the backend with the necessary headers.
